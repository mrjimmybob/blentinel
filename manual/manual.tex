\section{Project Overview}

Blentinel is a high-performance, multi-tenant monitoring platform designed for Managed Service Providers (MSPs). It enables a central IT organization to monitor distributed client networks, including those behind NAT or private firewalls, using a secure push-based architecture.

The system is composed of a Hub and multiple Probes. Probes operate inside client environments and push encrypted telemetry to the Hub. The Hub stores and presents monitoring data through a web interface.

% ============================================================
\section{Motivation and Design Philosophy}

Traditional monitoring solutions typically require inbound firewall rules, VPN connectivity, or persistent network tunnels. These approaches increase attack surface and operational complexity.

Blentinel is built around a strict outbound-only model:

\begin{itemize}
\item Probes initiate all connections.
\item No inbound firewall rules are required.
\item The Hub never connects back to probes.
\end{itemize}

This model improves security posture and simplifies deployment in zero-trust and MSP environments.

Key principles:

\begin{itemize}
\item Push-based communication
\item Cryptographic identity per probe
\item Explicit probe whitelisting
\item Defense-in-depth encryption model
\item Minimal operational complexity
\end{itemize}

\section{System Architecture}

Blentinel consists of three core components:

\subsection{Sentinel Probe (Rust)}

A low-footprint service deployed inside monitored networks. It:

\begin{itemize}
\item Polls local and network resources
\item Signs and encrypts reports
\item Pushes telemetry to the Hub
\end{itemize}

\subsection{Blind Sentinel Hub (Rust)}

The central coordination service responsible for:

\begin{itemize}
\item Authenticating probe identities
\item Verifying digital signatures
\item Decrypting reports
\item Storing time-series data in SQLite
\item Exposing a web interface
\end{itemize}

\subsection{Mobile Client (Planned)}

A future mobile application for technicians to view monitoring data.

\begin{longtable}{ll}

Frontend & Leptos \\
Transport Security & Optional TLS/HTTPS \\
Payload Encryption & X25519 + ChaCha20-Poly1305 \\
Authentication & Ed25519 Signatures \\
\end{longtable}

% ============================================================
\section{Security Model}

Blentinel implements multi-layer security:

\subsection{Application Layer}

\begin{itemize}
\item Ed25519 digital signatures (probe identity)
\item X25519 key exchange
\item ChaCha20-Poly1305 authenticated encryption
\end{itemize}

\subsection{Transport Layer (Optional)}

\begin{itemize}
\item TLS/HTTPS
\item Certificate pinning
\end{itemize}

\subsection{Zero-Trust Whitelisting}

The Hub rejects any probe whose public key is not explicitly listed in the configuration.

\section{Secure Bootstrap Procedure}

This section describes the correct first-time setup.

\subsection{Step 1 — Build and Deploy Hub}

\begin{lstlisting}
cargo build -p blentinelmake --release
./target/release/blentinelmake hub publish
\end{lstlisting}

Deploy the generated \texttt{publish/hub/app} directory.

Initialize configuration:

\begin{lstlisting}
./hub --init
\end{lstlisting}

Start the hub:

\begin{lstlisting}
./hub
\end{lstlisting}

On first run, the hub generates:

\begin{itemize}
\item hub\_identity.key
\item hub\_auth.token
\end{itemize}

These files must be protected.

\subsection{Step 2 — Build and Deploy Probe}

\begin{lstlisting}
./target/release/blentinelmake probe publish --target x86_64-unknown-linux-gnu
\end{lstlisting}

Deploy the \texttt{app} directory to the probe host.

Initialize configuration:

\begin{lstlisting}
./probe --init
\end{lstlisting}

Edit \texttt{blentinel\_probe.toml}.

\subsection{Step 3 — Probe Identity Generation}

On first run:

\begin{lstlisting}
./probe
\end{lstlisting}

The probe generates an Ed25519 keypair and prints its public key.

\subsection{Step 4 — Register Probe in Hub}

Edit \texttt{blentinel\_hub.toml}:

\begin{lstlisting}
[[probes]]
name = "HOMELAB"
public_key = "HEX_PUBLIC_KEY"
\end{lstlisting}

The hub hot-reloads this change.

\subsection{Correct Order}

\begin{enumerate}
\item Start Hub
\item Start Probe (generate key)
\item Add probe key to hub config
\item Restart probe
\end{enumerate}

\section{HTTPS Configuration}

Blentinel supports optional TLS.

Enable in \texttt{blentinel\_hub.toml}:

\begin{lstlisting}
[server.tls]
enabled = true
cert_path = "hub_tls_cert.pem"
key_path = "hub_tls_key.pem"
https_port = 3443
\end{lstlisting}

Certificates are auto-generated on first run.

Probes must:

\begin{itemize}
\item Copy hub\_tls\_cert.pem into probe build directory
\item Rebuild probe
\item Change hub\_url to https://
\end{itemize}

Certificate changes require probe rebuild.

\section{Building from Source}

\subsection{Using blentinelmake}

\begin{lstlisting}
cargo build -p blentinelmake --release
./target/release/blentinelmake
\end{lstlisting}

Interactive mode is available when run without arguments.

\subsection{Manual Build}

Hub:

\begin{lstlisting}
cargo leptos build --release
\end{lstlisting}

Probe:

\begin{lstlisting}
cargo build -p probe --release
\end{lstlisting}

\section{Cross Compilation}

The Hub cannot be cross-compiled due to Leptos constraints.

The Probe can be cross-compiled:

\begin{lstlisting}
rustup target add x86_64-unknown-linux-gnu
cargo build -p probe --release --target x86_64-unknown-linux-gnu
\end{lstlisting}

\section{Service Installation}

\subsection{Linux (systemd)}

Create:

\texttt{/etc/systemd/system/blentinel-probe.service}

\begin{lstlisting}
[Unit]
Description=Blentinel Probe
After=network.target

[Service]
ExecStart=/opt/blentinel/probe/probe
Restart=always
User=blentinel

[Install]
WantedBy=multi-user.target
\end{lstlisting}

Enable:

\begin{lstlisting}
sudo systemctl enable blentinel-probe
sudo systemctl start blentinel-probe
\end{lstlisting}

\subsection{Windows}

Use provided PowerShell installer script.

\section{Configuration Hot Reload}

Both Hub and Probe monitor configuration files.

\subsection{Probe — Fully Hot Reloadable}

All agent settings and resources can change without restart.

\subsection{Hub — Partially Hot Reloadable}

Hot-reloadable:
\begin{itemize}
\item Probe whitelist
\item probe\_timeout\_secs
\end{itemize}

Restart-required:
\begin{itemize}
\item server.host
\item server.port
\item db\_path
\item identity\_key\_path
\item auth\_token\_path
\end{itemize}

Invalid configs are rejected and previous config retained.

\section{Operational Behavior}

\subsection{Generated Runtime Files}

Hub:
\begin{itemize}
\item hub\_identity.key
\item hub\_auth.token
\item hub\_tls\_cert.pem
\item hub\_tls\_key.pem
\item blentinel.db
\end{itemize}

Probe:
\begin{itemize}
\item identity.key
\item blentinel\_blackbox.db
\end{itemize}

\subsection{BlackBox Reset Logic}

If probe identity or hub public key changes, the BlackBox database is automatically purged to prevent cryptographic mismatch errors.

% ============================================================
\section{Troubleshooting}

\subsection{Invalid Signature}

Occurs when probe public key does not match hub whitelist.

\subsection{Decryption Failed}

Usually indicates:

\begin{itemize}
\item Changed probe identity
\item Changed hub identity
\item Stale BlackBox database
\end{itemize}

Deleting \texttt{blentinel\_blackbox.db} forces a clean state.
